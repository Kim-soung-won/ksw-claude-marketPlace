# component-skill-manager — SKILL.md 포맷 규칙

> 이 파일은 `agents/component-skill-manager.md`에서 분리된 정적 참고 자료다.
> 본문(워크플로 로직)과 분리해 SKILL.md 작성 단계에 진입했을 때만 Read로 읽는다.
> 결과 리포트 출력 형식은 같은 디렉터리의 `result-output-format.md`를 참고한다.

이 규칙은 `@we/ai-template` 같은 **공용 React 컴포넌트 라이브러리**의 개별
컴포넌트(또는 모듈) 하나를 문서화하는 SKILL.md에 적용된다. 목표는 소비측
개발자가 "이 컴포넌트를 어떻게 import하고, 어떤 props로, 어떤 제약 아래에서
쓰는가"를 소스를 다시 열어보지 않고도 판단하게 하는 것이다.

---

## Frontmatter

```yaml
---
name: {스킬명}
description: >-
  {아래 "description 작성 규칙"에 따라 작성 — 트리거 + 설계 의도 + 핵심 제약}
---
```

- **frontmatter는 `name`과 `description` 두 필드만 사용한다.** 컴포넌트 스킬은
  도메인 스킬과 달리 `metadata`/`confidence` 블록을 두지 않는다. 확신하지 못하는
  내용은 body에 `<!-- unconfirmed: ... -->` 주석으로 표시하고, 원칙적으로는
  소스에서 확인한 사실만 기술한다.

### `name` 규칙

- 컴포넌트 skill의 `name`은 **PascalCase**다 (도메인 skill의 kebab-case와 다르다).
- 카테고리가 있는 컴포넌트는 **`{Category}_{ComponentName}`** 형식을 쓴다.
  - 예: `Button_CopyButton`, `Card_StatusCard`, `Chart_BarChart`, `Badge_StatusBadge`
- 카테고리 없이 단독 모듈이거나 여러 variant를 묶는 우산 스킬이면 카테고리명
  하나만 쓴다.
  - 예: `Card`, `Input`, `Table`, `Provider`, `Toast`
- **폴더명 = frontmatter `name` = export되는 컴포넌트/카테고리명**, 세 가지가
  반드시 일치해야 한다. 오타 패턴(`Buton`, `Compnent` 등)이 의심되면 생성 전에
  사용자에게 확인한다.

### `description` 작성 규칙

> **description은 스킬 자동 호출의 트리거 조건이다.**
> Claude는 description을 읽고 "지금 이 스킬을 호출해야 하는가?"를 판단한다.

컴포넌트 스킬의 description은 다음 세 요소를 순서대로 담는다:

1. **트리거 문장** — 언제 이 문서를 먼저 읽어야 하는가.
   `"{컴포넌트}(또는 모듈)를 사용·수정·통합하는 작업 시 이 문서를 먼저 읽는다.
   \"@we/ai-template\"에서 {컴포넌트}를 import하거나 관련 파일을 변경하는 작업에
   트리거한다."` 를 기본형으로 한다. 대체 이름·별칭(예: `MessageInput`,
   `TabBar`)이 있으면 키워드로 함께 나열해 트리거 적중률을 높인다.
2. **설계 의도** — 이 컴포넌트가 왜 존재하고 무엇을 공통화하는가. body의
   `## 설계 의도` 첫 문단과 일치시킨다.
3. **핵심 제약(있을 때)** — Provider 필요, 특정 버전 이상, 허용 prop 값 유니온
   제한 등 어기면 런타임 오류·미동작으로 이어지는 제약을 한두 문장으로 요약한다.

예시(짧은 형태):

```
CopyButton 컴포넌트를 사용·수정·통합하는 작업 시 이 문서를 먼저 읽는다.
"@we/ai-template"에서 CopyButton을 import하거나 관련 파일을 변경하는 작업에
트리거한다. 설계 의도: 텍스트를 클릭 한 번으로 클립보드에 복사하고 토스트로
결과를 피드백하는 아이콘 버튼이다. useToast() 연동이 내부에 캡슐화돼 있다.
```

예시(제약 강조 형태):

```
Alert 컴포넌트를 다룰 때 읽는다. CustomAlert, useAlert, alertManager 키워드가
등장하거나 window.confirm·인라인 모달을 교체할 때 트리거한다. 핵심 제약:
CustomAlert을 직접 렌더링하지 않고 반드시 useAlert().show()로 호출해야 하며,
onConfirm 내부에서 hide()를 수동 호출하면 퇴장 애니메이션이 깨지므로 금지한다.
WeAiTemplateProvider가 앱 루트에 마운트돼 있어야 동작한다.
```

## 바디 섹션 구조

컴포넌트 스킬은 아래 6개 섹션을 **이 순서**로 구성한다.

| 섹션 | 필수/선택 | 포함 기준 |
|------|-----------|-----------|
| `## 설계 의도` | **필수** | 항상. 왜 존재하는가 + 무엇을 공통화·캡슐화하는가 |
| `## Interface` | **필수** | import 경로, export 목록, props 타입 |
| `## 올바른 사용` | **필수** | 대표 사용 패턴을 tsx 코드 블록으로 |
| `## 의도되지 않은 사용` | **필수** | 안티패턴 ❌ / 올바른 대안 ✅ 쌍으로 |
| `## 마이그레이션 범위` | 선택 | 기존 인라인 구현을 이 컴포넌트로 교체하는 작업일 때 |
| `## 마이그레이션 체크리스트` | 선택 | `마이그레이션 범위`를 포함했을 때 함께 |

여러 variant를 묶는 우산 스킬(예: `Card`)은 `## 설계 의도` 아래에 **variant 선택
기준 표**(Variant / 목적 / 언제 선택하는가)를 둔다.

### `## 설계 의도`

- 컴포넌트가 해결하는 반복 문제와, 그것을 어떻게 캡슐화했는지 1~2문단으로 기술한다.
- 소비측이 더 이상 직접 작성하지 않아도 되는 로직(클립보드 처리, 토스트 연동,
  권한 분기 등)을 명시한다.

### `## Interface`

- **반드시 실제 소스에서 추출한 사실만 적는다.** import 경로, 컴포넌트/타입 export
  목록, props의 실제 타입을 기재한다.
- 형태 A — 단일 컴포넌트: props 시그니처를 TypeScript로 직접 보여준다.

  ````markdown
  ```typescript
  // copy-button.tsx
  export function CopyButton({ content }: { content: string }): JSX.Element
  ```
  ````

- 형태 B — 여러 export를 묶는 모듈: import 블록 + export 표(Export / 종류 / 파일).

  | Export | 종류 | 파일 |
  |--------|------|------|
  | `withCard` | HOC 함수 | `with-card.tsx` |
  | `StatusCardProps` | interface | `StatusCard.tsx` |

- prop별 상세가 필요하면 props 표(prop / 타입 / 필수 / 기본값 / 설명)를 추가한다.
- 허용값이 유니온인 prop(`color`, `status`, `type` 등)은 **가능한 값 전체를 나열**
  한다. 나열 밖의 값은 스타일/동작이 없어 깨진다는 점을 함께 적는다.

### `## 올바른 사용`

- 가장 흔한 사용 시나리오를 tsx 코드 블록으로 2개 이상 제시한다.
- 우산 스킬이면 "상황 → 올바른 선택" 표를 앞에 두고 이어서 코드 예시를 든다.

### `## 의도되지 않은 사용`

- 실제로 발생하는 오용을 **굵은 제목 + ❌ 안티패턴 / ✅ 올바른 대안** 쌍으로 적는다.
- 각 쌍에는 "왜 안 되는가"를 주석으로 한 줄 붙인다(런타임 오류, 타입 오류, 스타일
  충돌, 필수 props 강제 등).
- 대표 오용 카테고리: Provider 누락, 허용 밖 prop 값, 존재하지 않는 prop 전달,
  내부 CSS 클래스 직접 오버라이드, variant 용도 혼용, 콜백 미지원인데 결과 처리 시도.

### `## 마이그레이션 범위`

교체형 도입 작업일 때만 포함한다. 세 블록으로 구성한다:

```
**수정 대상 파일 패턴**
(인라인 구현이 존재할 만한 소비측 glob)

**수정 금지 파일 패턴**
(컴포넌트/스타일 본체 — 소비측에서 직접 수정하지 않는다)

**사전 확인 사항**
(라이브러리 버전, Provider 존재, 함께 import할 타입 등)
```

### `## 마이그레이션 체크리스트`

- 도입 완료 기준을 `- [ ]` 체크박스로 나열한다.
- 마지막 항목은 항상 "기존 동작 대비 시각적·기능적 회귀가 없다"로 마무리한다.

## 작성 지침

1. **소스가 곧 근거다** — Interface(props·타입·export)는 반드시 실제 컴포넌트
   소스를 Read로 읽어 추출한다. 소스를 확인하지 못한 항목은 추측으로 채우지 않고
   `<!-- unconfirmed: {무엇을 확인해야 하는가} -->` 주석으로 표시한다.

2. **추측 금지** — 존재 여부가 불확실한 prop·콜백·기본값을 지어내지 않는다.
   특히 `onSuccess`/`className` 같은 "있을 법한" prop을 실제 확인 없이 예시에
   넣지 않는다. 오용 예시에서 "이 prop은 없다"를 명시하는 것은 권장된다.

3. **한국어 + 영어 식별자 병기** — 설명은 한국어로, 컴포넌트·prop·타입 이름은
   코드의 영어 식별자를 그대로 병기한다.

4. **제약은 반드시 근거와 함께** — "Provider 필요", "특정 버전 이상", "prop 값
   유니온 제한" 같은 제약은 왜 그런지(내부에서 무엇을 호출/참조하는지)를 한 줄로
   덧붙인다.

5. **UPDATE 시 기존 내용 보존** — 기존 섹션을 통째로 삭제하지 않는다. 변경된
   부분만 수정하거나 새 정보를 추가한다. 라이브러리 버전이 올라 prop이 바뀐
   경우 변경 전/후를 함께 남긴다.

6. **내부 CSS 클래스 직접 스타일링 경고** — 컴포넌트가 `__badge`, `__copy-btn`
   같은 내부 클래스를 쓴다면, 소비측이 그 클래스에 직접 스타일을 주입하면
   업그레이드 시 깨진다는 점을 "의도되지 않은 사용"에 반드시 포함한다.

## 자가 검증 체크리스트

SKILL.md 작성 완료 후 다음을 순서대로 확인한다. 결과 리포트에 PASS/FAIL 수를
포함한다.

### Frontmatter 검증

- [ ] `name`이 폴더명과 정확히 일치하는가?
- [ ] `name`이 PascalCase이며, 카테고리가 있으면 `{Category}_{ComponentName}` 형식인가?
- [ ] `description`에 트리거 문장 + 설계 의도가 모두 포함돼 있는가?
- [ ] 어기면 런타임 오류·미동작을 유발하는 핵심 제약이 있다면 description에 요약돼 있는가?
- [ ] `description`의 설계 의도가 body `## 설계 의도` 첫 문단과 일치하는가?

### 이름 검증

- [ ] 컴포넌트/카테고리명에 오타 패턴이 없는가?
- [ ] 폴더명 = frontmatter `name` = 실제 export명, 모두 일치하는가?

### Body 검증

- [ ] `## 설계 의도`, `## Interface`, `## 올바른 사용`, `## 의도되지 않은 사용` 4개 필수 섹션이 모두 존재하는가?
- [ ] `## Interface`의 import 경로·props·export가 실제 소스에서 추출한 값인가? (추측 아님)
- [ ] 유니온 타입 prop(`color`/`status`/`type` 등)의 허용값을 전부 나열했는가?
- [ ] `## 올바른 사용`에 tsx 코드 예시가 2개 이상 있는가?
- [ ] `## 의도되지 않은 사용`이 ❌ 안티패턴 / ✅ 대안 쌍으로 작성됐고, 각 쌍에 "왜 안 되는가"가 붙어 있는가?
- [ ] 소스에서 확인하지 못한 항목이 `<!-- unconfirmed -->`로 표시돼 있고, 추측으로 채운 prop·콜백이 없는가?
- [ ] 우산(variant) 스킬이라면 variant 선택 기준 표가 있는가?

### 마이그레이션 검증 (해당 시)

- [ ] `## 마이그레이션 범위`에 수정 대상/금지 파일 패턴과 사전 확인 사항이 모두 있는가?
- [ ] 컴포넌트 본체(라이브러리 소스)가 "수정 금지 파일 패턴"에 포함돼 있는가?
- [ ] `## 마이그레이션 체크리스트` 마지막 항목이 "기존 동작 대비 회귀 없음"인가?

### UPDATE 모드 검증 (해당 시)

- [ ] 기존 섹션이 삭제되지 않고 보존됐는가?
- [ ] prop·타입 변경이 있다면 변경 전/후가 함께 기록됐는가?
