# 단위 테스트 작성 규약 (vitest)

도메인 스킬과 함수 계약으로부터 **"코드가 하는 것"이 아니라 "해야 하는 것"을**
테스트하기 위한 규약이다. React + rsbuild + Module Federation, zustand +
react-query, FSD 구조 프로젝트에서 검증됐다.

> 이 문서는 **규약(지식)**이다. 실제 작성·실행은 `test-writer` 에이전트가 이
> skill을 로드해 따르고, 작성된 테스트의 감사(change-detector·약한 단언 적발)는
> `test-reviewer` 에이전트가 맡는다.
>
> 📎 실제 프로젝트에 커밋된 골드 예시(스킬 기반 스토어 테스트 / 계약 기반 검증
> 함수 테스트)는 같은 폴더의 [`examples.md`](./examples.md)에 있다. 단언을 어디서
> 어떻게 끌어올지 판단이 서지 않을 때 읽는다.

## 최우선 원칙: 단언의 출처를 이원화한다

> **이 문서에서 가장 중요한 규칙이다.** 무엇을 단언할지는 구현 코드에서 베끼는
> 것이 아니라, 아래 두 출처 중 하나에서 **유도**한다. 구현을 그대로 박제하면
> 리팩터링마다 깨지는 change-detector가 되어 테스트의 가치가 사라진다.

테스트할 모듈을 먼저 두 갈래로 분류한다.

```
이 모듈에 대응하는 도메인 스킬(_scenario/_api/_domain)이 있나?
├─ 있다 (도메인 행위 모듈) ......... 스킬(명세)에서 단언을 유도  → 아래 A
└─ 없다 (순수 로직·유틸·검증) ...... 함수 계약에서 단언을 유도    → 아래 B
```

### A. 스킬 기반 — 도메인 행위가 있는 모듈

프로젝트의 도메인 스킬에서 기대 동작을 끌어온다. 특히 `*_scenario.md`의
**given/when/then · 상태 전이 · "핵심 제약사항"**이 테스트 케이스로 거의 1:1
매핑된다. `*_api.md`(입출력 계약)와 `*_domain.md`(도메인 규칙)도 함께 참조한다.

- 각 테스트 케이스 주석에 근거가 된 스킬 항목을 남긴다
  (예: `// scenario: "상태 관리 흐름" — step 0→1 전이`). 나중에 스펙이 바뀌면
  어느 테스트를 고쳐야 하는지 추적 가능해진다.
- **선행 게이트 — 낡은 스킬 = 낡은 테스트.** 스킬 기반 테스트를 쓰기 전에, 그
  도메인 스킬이 현재 코드와 어긋나지 않았는지 `domain-skill-reviewer`로 신뢰도를
  먼저 확인한다. 스킬이 낡았으면 테스트도 낡은 스펙을 박제한다.
- **실패는 노이즈가 아니라 스펙-구현 정합성 시그널이다.** 스킬에서 유도한 테스트가
  구현과 어긋나 실패하면, 둘 중 하나가 틀린 것이다 — 코드가 스펙을 어겼거나,
  스킬이 낡았거나. 어느 쪽인지는 사람이 판정한다. 테스트를 통과시키려고 단언을
  구현에 맞춰 조용히 고치지 않는다.

### B. 계약 기반 — 대응 스킬이 없는 순수 로직

알고리즘 유틸·검증 함수처럼 대응 스킬이 없는 모듈은 함수의 **계약**에서 유도한다.

- **무게를 싣는 곳**: 시그니처, 반환 타입, JSDoc·코드에 명문화된 규칙, 타입·구조
  **불변식**(예: `isValid ⇔ errors.length === 0`).
- **최소화하는 곳**: 에러 **문자열 정확 일치**, private 분기의 세부 구현 —
  이런 단언은 계약 기반에서도 change-detector가 되는 위험 지점이다. 문자열은
  전체 일치 대신 핵심 부분 포함(`.includes(...)`) 수준으로, 문서화된 규칙에
  묶여 있을 때만 쓴다.

## 배치 컨벤션

루트 `test/`가 `src/`를 **미러링**한다.

```
src/shared/store/mcp/server-id-store.ts
  → test/shared/store/mcp/server-id-store.test.ts

src/entities/workflow/workflow/node-validator.ts
  → test/entities/workflow/workflow/node-validator.test.ts
```

프로젝트에는 이미 다음이 세팅돼 있다(새로 만들지 않는다).

- `vitest.config.ts`: `@/` → `src` alias, `test/**/*.{test,spec}.{ts,tsx}` 수집,
  기본 `environment: "node"`.
- `tsconfig.test.json`: `test`와 `vitest.config.ts`를 포함해 테스트 범위 타입 체크.

## 실행 환경 선택 — node vs jsdom

| 대상 | 환경 | 방법 |
|------|------|------|
| 순수 함수, 검증 로직 | node (기본) | 추가 설정 없음. DOM 불필요 |
| zustand 스토어 | node (기본) | `getState()`/`setState()`로 상태 검증. 렌더 불필요 |
| RTL 컴포넌트/훅 렌더 | jsdom | 파일 **상단**에 `// @vitest-environment jsdom` |

### RTL 주의 — 공용 렌더 하니스가 전제다

컴포넌트/훅 렌더 테스트는 `QueryClientProvider` · `LumenProvider` ·
`ToastProvider` · i18n · Module Federation 컨텍스트를 감싼 **공용 test-utils
렌더 하니스**로 렌더한다. 이 하니스는 모든 RTL 테스트가 공유하는 전제이므로,
순수 함수/스토어 테스트와 달리 "완전 독립·병렬"이 아니다. Provider를 매 파일에서
직접 조립하지 말고 공용 하니스를 경유한다.

## 검증 규율 — 작성으로 끝내지 않는다

테스트는 **작성 + 통과 확인**까지가 완료다. validate-fix-repeat로 진행한다.

1. 작성한 파일을 `npx vitest run <파일경로>`로 실행한다.
2. 실패하면 원인을 판정한다 — 테스트가 틀렸나, 스펙(스킬)이 낡았나, 코드가
   틀렸나. **테스트를 통과시키려고 단언을 무르게 고치지 않는다.**
3. 스킬 기반 테스트의 실패는 스펙-구현 정합성 시그널이므로, 조용히 넘기지 말고
   판정 결과를 사람에게 보고한다.
4. 타입은 `tsconfig.test.json` 범위에서 확인된다.

## 워크플로 체크리스트

응답에 이 체크리스트를 복사해 진행 상황을 추적한다.

```
테스트 작성 진행:
- [ ] 1. 대상 모듈 분류: 도메인 행위 모듈(A) / 순수 로직(B)
- [ ] 2. (A인 경우) 대응 도메인 스킬을 domain-skill-reviewer로 신뢰도 확인 (선행 게이트)
- [ ] 3. 단언 출처 수집: (A) scenario given/when/then·상태 전이·핵심 제약 / (B) 시그니처·JSDoc·불변식
- [ ] 4. 배치 경로 결정: test/가 src/를 미러링
- [ ] 5. 환경 선택: 순수 로직·스토어=node / RTL=jsdom + 공용 하니스
- [ ] 6. 테스트 작성 (각 케이스에 근거 주석)
- [ ] 7. npx vitest run <파일>로 통과 확인 (validate-fix-repeat)
- [ ] 8. 실패 시 스펙-구현 정합성 판정 후 보고 (단언을 무르게 고치지 않기)
```

## 진행 톤

- 단언은 항상 **명세(스킬) 또는 계약**에서 유도한다. 구현 코드를 그대로 베끼지
  않는다(change-detector 금지).
- 스킬 기반 테스트 전에는 **스킬 신뢰도를 먼저** 확인한다(낡은 스킬 = 낡은 테스트).
- 통과 확인 없이 완료로 보지 않는다. 실패는 판정 대상이지 통과시켜야 할 방해물이
  아니다.
