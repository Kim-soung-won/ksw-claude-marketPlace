# test-reviewer 감사 체크리스트 (a)~(e)

각 테스트 파일에 대해 아래 항목을 순서대로 점검한다. 판정 근거는 항상
`${CLAUDE_PLUGIN_ROOT}/skills/test-authoring/SKILL.md`의 규약이며, 소스·테스트에서
확인되지 않은 것을 단정하지 않는다.

---

## (a) 단언이 구현이 아니라 명세/계약에서 나왔는가

- 대응 도메인 스킬(`.claude/skills/*_scenario|_api|_domain`)이 있으면, 단언이 그 명세의
  시나리오·API 계약·불변식과 대응하는지 대조한다.
- 대응 스킬이 없으면 함수 계약(시그니처/JSDoc/반환 불변식)과 대조한다.
- 구현 코드의 내부 계산 결과를 그대로 기대값으로 박아둔 **change-detector** 단언인지 본다.
  (예: 구현이 반환하는 값을 그대로 복사해 `toEqual`한 것 — 명세 근거 없음)

```bash
# 대응 도메인 스킬 존재 여부
find "$(git rev-parse --show-toplevel)/.claude/skills" -name "SKILL.md" | sort
```

- 명세·계약 근거 없이 구현 결과만 박제 → **CRITICAL** (박제, 신뢰 불가)

## (b) 깨지기 쉬운 구현 결합 단언

- 에러 **문자열 정확 일치**(`toThrow('정확한 메시지')`), 로그 문구, 내부 호출 순서,
  private 구현 세부에 결합된 단언을 찾는다.
- 이런 단언은 리팩터링만 해도 깨지므로 불변식·에러 타입/코드 기반으로 완화해야 한다.

- 구현 결합 단언 존재 → **WARNING** (완화 필요, 어떻게 바꿀지 명시)

## (c) 불변식 누락

- 명세/계약이 요구하는 불변식(상태 일관성, 경계값, 멱등성 등)이 단언에서 빠졌는지 본다.
- change-detector일수록 "값 일치"만 있고 "지켜야 할 성질"은 비어 있는 경향이 있다.

- 핵심 불변식이 빠짐 → **WARNING** (어떤 불변식을 추가해야 하는지 명시)

## (d) test/ 미러 경로·명명 규약 준수

- `src/x/y.ts` → `test/x/y.test.ts` 미러 규약을 지켰는지 확인한다(정확한 규칙은 스킬).

```bash
# 미러 대응이 어긋난 테스트 파일 탐지의 출발점
find "$(git rev-parse --show-toplevel)/test" -name "*.test.ts" | sort
```

- 경로·명명 규약 위반 → **WARNING**

## (e) 작성 후 실제 통과하는지

```bash
npx vitest run <대상 test 파일>
```

- 지금 실패하는 테스트(깨진 채 방치) → **CRITICAL**
- 통과하지만 위 (a)~(d) 문제가 있는 경우는 각 항목 등급을 따른다.

---

## 등급 기준 요약

| 등급 | 아이콘 | 기준 |
|------|--------|------|
| CRITICAL | 🔴 | 이 테스트를 믿으면 안 되는 상태 — 명세 근거 없는 구현 박제, 현재 실패 |
| WARNING | 🟡 | 신뢰도 저하 — 구현 결합 단언, 불변식 누락, 경로·명명 위반 |
| INFO | 🟢 | 개선 권장 — 사소한 스타일·중복, 방치해도 되지만 고치면 좋은 것 |

- 근거 없는 추측성 이슈는 등록하지 않는다.
- 도메인 스킬 자체의 신뢰도 의심은 이 에이전트가 판정하지 않고 domain-skill-reviewer 선행
  권고로 남긴다.
