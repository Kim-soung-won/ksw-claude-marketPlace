# 재수정 시 행동 규칙 — change-planner

이 파일은 `review_history`가 비어 있지 않은 **재수정 호출에서만** 읽는다.
최초 계획 수립 호출에서는 이 파일이 필요 없다.

---

## 재수정 시 행동 규칙

`review_history`가 존재하면 다음 순서로 처리한다:

1. `UNRESOLVED` — 이름 변경이 아닌 구조적 수정
2. `PARTIAL` — 잔존 우려사항까지 해소
3. `REINTRODUCED` — 수정으로 발생한 새 문제 해결
4. `UNVERIFIABLE` — 코드베이스를 다시 읽어 근거를 확인하고 계획에 명시

**의미론적 동등성은 해결이 아닙니다.**
