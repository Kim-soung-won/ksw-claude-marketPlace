# 위키링크 연계 절차

Obsidian vault(`~/agent-knowledge-base`)에서 위키링크(`[[...]]`)가 0개인
콘텐츠 노트를 찾아, 다른 노트와의 실제 개념적 연관성을 근거로 `## Related Notes` 섹션에
단방향 위키링크를 추가하는 절차. `note-capture.md` 절차의 마지막 단계에서 이어서
호출되거나, 독립적인 링크 보강/점검 요청으로 직접 호출된다.

---

## 탐지 커맨드

```bash
cd ~/agent-knowledge-base
find . -name "*.md" -not -path "./.git/*" -not -path "./.obsidian/*" \
  -not -path "./_templates/*" -not -path "./00 Meta/*" -not -path "./.claude/*" \
  | sort > /tmp/kb_all_notes.txt
tr '\n' '\0' < /tmp/kb_all_notes.txt | xargs -0 grep -L '\[\[' | sort
```

`CLAUDE.md`가 결과에 포함되면 콘텐츠 노트가 아니므로 대상에서 제외한다(vault 루트 시스템
설정 파일). 코드 블록 안의 배열 리터럴(`[[ {...} ]]`)이 위키링크로 오탐될 수 있으니
탐지된 파일은 실제로 링크가 없는지 본문을 확인한다.

---

## 링크 판단 기준 (Linking Guide)

1. **탐색 순서**: 같은 폴더 우선 → 카탈로그의 tags/category 겹침 → 본문 키워드/개념
2. **삽입 기준**: 본문에서 다른 노트의 핵심 개념을 실제로 언급하거나 전제로 삼는 경우에만
   링크한다. "같은 태그" 같은 이유만으로 걸지 않는다 — 억지 연결 금지. 연관 노트가 없으면
   무리해서 채우지 않는다(개수를 채우는 것이 목표가 아니다).
3. **개수 제한**: 노트 1개당 신규 링크 최대 5개, 동일 대상 중복 금지.
4. **파일명이 불명확한 경우**: `[[폴더 경로/파일명|표시명]]` 형태로 폴더 경로를 포함한
   전체 경로 + alias를 사용해 대상 개념을 드러낸다. 같은 폴더 안에서는 짧게 `[[파일명|표시명]]`.
5. **frontmatter는 절대 수정하지 않는다.** 본문 최하단에 다음 형식으로 추가(이미 섹션이
   있으면 그 아래에 append):

   ```
   ---

   ## Related Notes

   - [[대상 노트]] — 연관 이유 1줄
   - [[대상 노트2|표시명]] — 연관 이유 1줄
   ```

---

## 처리 절차

### 대상이 신규/수정 노트 1~2개인 경우 (소규모)
직접 처리한다:
1. 대상 노트를 Read로 읽는다.
2. 같은 폴더의 다른 노트를 Grep/Read로 탐색해 관련 후보를 찾는다.
3. 위 판단 기준에 따라 링크를 삽입한다.
4. 아래 "검증"을 실행한다.

노트 수가 5개 미만이면 전수 검토가 자동으로 성립하므로 별도 표본 규칙이 필요 없다.

### 대상이 5개 이상이거나 폴더 전체 재점검인 경우 (대규모)
1. vault 전체 콘텐츠 노트의 (제목/경로/태그/1줄 요약) 카탈로그를 구성한다.
2. 대상 노트를 폴더 단위로 배치를 나눈다.
3. 배치별로 nested Subagent(general-purpose)를 병렬 호출한다. 각 Subagent는:
   - 자기 배치에 속한 파일만 Edit 가능 (다른 배치 파일 수정 금지)
   - 링크는 항상 단방향 — 자기 노트에 다른 노트로의 링크를 추가할 뿐, 대상 노트를 열어
     역방향 링크를 쓰지 않는다. 배치가 폴더 경계로 배타적으로 분할되어 있으면 동시 쓰기
     충돌이 구조적으로 발생하지 않는다.
   - 카탈로그 전체(자기 배치 밖 노트 포함)를 링크 후보로 사용할 수 있다.
4. 삽입된 링크를 검증한다(아래 "검증" 참조).
   - 노트 수가 43개 이하로 적을 때: 전수 검토(모든 노트, 모든 링크를 직접 대조)를 기본값으로 한다.
   - 노트 수가 크게 늘어나는 향후 시나리오: 삽입된 전체 링크의 최소 30% 또는 배치 노트
     수 중 큰 쪽을 샘플링하고, 표본 중 부적절 판정이 15%를 초과하면 해당 배치를 반려하고
     재작업한다.

---

## 검증

1. **구조 검증**: 모든 `[[...]]` 링크의 대상 파일이 실제로 존재하는지, alias 문법이
   올바른지, 노트당 5개를 초과하지 않는지, 동일 대상 중복이 없는지 확인한다.
2. **frontmatter 무결성**: `git show HEAD:<path>`와 현재 파일의 frontmatter 블록을
   대조해 완전히 동일한지 확인한다.
3. **내용 검토**: 삽입된 각 링크의 근거 문장이 실제로 두 노트의 본문 내용과 부합하는지
   확인한다. 부적절한 링크(단순 키워드 일치, 억지 연결)는 제거하거나 재작업한다.
