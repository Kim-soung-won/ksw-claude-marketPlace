# Subagent 작성 모범 사례: description / 프롬프트 / 도구 스코핑

> 이 문서는 공식 문서(`subagent-docs.md`)를 보충하는 저장소 자체 가이드다. 공식
> 문서가 "필드가 무엇인지"를 정의한다면, 이 문서는 "그 필드를 어떻게 채워야
> 라우팅·실행 품질이 올라가는지"를 다룬다. `subagent-docs.md`에 없는 frontmatter
> 필드나 동작을 여기서 새로 만들어내지 않는다 — 필드 스펙은 항상 공식 문서가
> 기준이다.

## 1. description: 라벨이 아니라 위임 조건(Delegation Condition)

`description`은 사람이 읽는 설명문이 아니라, 메인 에이전트가 사용자 요청을 분석한
뒤 "이 subagent를 스폰할지"를 결정하는 **자동 라우팅 트리거**다. 서브에이전트를
"독립된 작업자에게 하청 주는 상황"으로 보고 다음 기준으로 작성한다.

- **금지**: "코드 리뷰 에이전트", "데이터 분석기" 같은 라벨형 한 줄 설명.
- **필수**: 언제 위임해야 하는지 조건문 형태로 명시한다.
  - "코드 변경 직후 선제적으로 코드 품질을 검토해야 할 때 사용하라"
  - "사용자가 쿼리 튜닝이나 성능 최적화를 요청할 때 이 에이전트에게 위임하라"
- **선제적 위임이 필요하면** "PROACTIVELY(선제적으로)" 또는 "use immediately
  after..." 같은 문구를 명시한다. 이 키워드가 있어야 Claude가 사용자의 명시적
  요청 없이도 알아서 백그라운드로 호출한다(공식 문서 "자동 위임 이해" 절 참고).
- 트리거 조건은 **행동/시점 기준**으로 쓴다. "무엇에 대한 에이전트인가"가 아니라
  "어떤 사용자 발화·어떤 작업 직후·어떤 상태일 때 호출되는가"를 적는다.
- 가능하면 example 블록(공식 문서 예제 subagent들처럼 `<example>` 태그 또는
  Context/user/assistant 3줄 구성)을 1~2개 포함해, 실제 트리거 문장과 위임
  결정을 구체적으로 보여준다.

체크리스트 (description 검토 시):

- [ ] "~할 때 사용하라/위임하라" 형태의 조건문이 있는가
- [ ] 라벨 한 줄로 끝나지 않았는가
- [ ] 선제적 호출이 필요한 경우 PROACTIVELY류 키워드가 있는가
- [ ] 실제 호출 시점을 보여주는 example이 있는가

## 2. 프롬프트(시스템 프롬프트) 작성 규칙: 단일 책임 + 출력 형식 강제

서브에이전트는 독립된 컨텍스트 창에서 혼자 작업한 뒤 탐색 과정(파일 검색 기록,
디버깅 로그 등)은 모두 버리고 **최종 결과만** 메인 에이전트에게 반환한다. 이
특성 때문에 프롬프트는 두 가지를 반드시 강제해야 한다.

### 2-1. 단일 책임 설계 (Single Responsibility)

- 하나의 subagent에 여러 개의 포괄적 역할을 섞지 않는다. "보안 감사", "테스트
  실행기"처럼 하나의 명확한 책임만 부여한다.
- 역할이 두 개 이상 필요해 보이면 subagent를 쪼개고, 오케스트레이터(메인
  에이전트 또는 상위 subagent)가 체이닝하도록 설계한다 — 공식 문서의 "Subagent
  체인" 패턴 참고.
- 판단 기준: 프롬프트에 "그리고", "또한"으로 이어지는 서로 다른 성격의 작업이
  나열되면 분리를 검토한다.

### 2-2. 반환 형식의 엄격한 정의 (Strict Output Contract)

메인 에이전트는 subagent의 사고 과정을 볼 수 없고 최종 텍스트만 받는다. 따라서
프롬프트 말미에 출력 형식을 구체적으로 못박아야 메인 에이전트가 즉시 활용할 수
있다.

- 나쁜 예: "결과를 보고하라"
- 좋은 예: "발견한 취약점을 HIGH/MEDIUM/LOW 등급으로 분류하여 마크다운
  리스트로만 반환하라. 탐색 과정이나 파일 목록은 출력하지 마라."
- 형식 지정 시 포함하면 좋은 요소:
  - 출력 스키마(마크다운 리스트, JSON, 표 등 구체적 형태)
  - 분류/정렬 기준(심각도, 우선순위, 카테고리)
  - 탐색 범위(어디까지 찾아볼지) 또는 탐색 범위 제외 대상
  - 제외할 내용("중간 과정 로그는 포함하지 않는다")
- 공식 문서 예제(코드 검토자, 디버거)의 "우선순위별 정리"나 "Root cause →
  Evidence → Fix → Test → Prevention" 같은 고정 구조를 참고해, 반환 형식을
  본문 하단에 명시적 섹션으로 적는다.

체크리스트 (프롬프트 검토 시):

- [ ] 이 subagent가 담당하는 책임이 한 문장으로 요약되는가
- [ ] 서로 다른 성격의 작업이 하나의 프롬프트에 섞여 있지 않은가
- [ ] 출력 형식(구조/등급/포맷)이 명시되어 있는가
- [ ] 메인 에이전트가 추가 파싱 없이 바로 쓸 수 있는 형태인가

## 3. 도구 권한 제한 (Tool Scoping)

목적에 맞는 도구만 허용(`tools`)하거나 불필요한 도구를 차단(`disallowedTools`)
한다. 필드 동작 자체(둘 다 지정 시 우선순위, MCP 패턴 등)는 `subagent-docs.md`의
"사용 가능한 도구" 절을 따른다 — 여기서는 **어떤 기준으로 스코핑할지**만 다룬다.

- **분석/탐색 전용 subagent** (코드베이스 구조 파악, 리뷰, 진단 등 결과만
  보고하고 수정하지 않는 역할)는 `Write`, `Edit`, `NotebookEdit`처럼 상태를
  바꾸는 도구를 반드시 제외한다. 읽기 전용으로 제한하면:
  - 서브에이전트가 검토 대상 코드를 실수로 고치는 사고를 원천 차단한다.
  - 메인 에이전트가 "이 결과는 관찰만 했다"고 신뢰할 수 있어 후속 판단이
    쉬워진다.
- **수정 작업이 필요한 subagent**(디버거, 리팩터러 등)만 `Edit`/`Write`를
  허용한다. 이 경우에도 `Bash`처럼 광범위한 도구는 필요성이 명확할 때만 포함한다.
- **외부 시스템 접근이 필요 없는 subagent**는 `mcp__*` 패턴을
  `disallowedTools`에 추가하거나 애초에 `tools` 허용 목록에서 제외해 MCP 도구
  전체를 차단한다.
- **판단 순서**: (1) 이 subagent가 코드/데이터를 변경해야 하는가? 아니면
  관찰·판단만 하는가? → (2) 변경이 필요 없다면 `tools`를 읽기 전용 집합
  (`Read, Grep, Glob` 등)으로 명시적으로 좁힌다. → (3) 변경이 필요하다면 어떤
  종류의 변경인지(파일 편집만? 셸 명령까지 필요?)에 따라 최소 집합을 고른다.
- 도구를 넓게 열어두는 것(필드 생략 → 전체 상속)은 "일단 다 되게" 만드는
  방법처럼 보이지만, 목적이 분명한 subagent일수록 도구를 좁히는 쪽이 성능과
  안전성 모두에 유리하다 — 이 판단을 스킵하지 않는다.

체크리스트 (도구 스코핑 검토 시):

- [ ] 이 subagent는 파일/상태를 변경해야 하는 역할인가, 관찰만 하는 역할인가
- [ ] 관찰 전용이라면 Write/Edit/NotebookEdit이 제외되어 있는가
- [ ] 이 subagent에 불필요한 MCP 서버 접근이 열려 있지 않은가
- [ ] `tools`를 생략(전체 상속)한 것이 의도적 선택인지, 검토 누락인지 구분되는가
